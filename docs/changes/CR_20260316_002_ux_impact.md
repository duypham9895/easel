# CR_20260316_002 — UX Impact Assessment

## Change Summary

Rebuild the Calendar screen from a mixed-purpose view (calendar grid + inline log panel + cycle stats card) into a focused Cycle Timeline with two-tap interaction: first tap marks a period day, second tap opens an edit bottom sheet. Remove MyCycleCard entirely.

---

| Screen | File | Impact Level |
|--------|------|--------------|
| Calendar tab (Moon) | `app/app/(tabs)/calendar.tsx` | **Major** — layout rewrite, tap logic rewrite |
| PeriodLogPanel | `app/components/moon/PeriodLogPanel.tsx` | **Major** — convert from inline `Animated.View` to `Modal` bottom sheet |

## 2. Component-Level Changes

### REMOVE: MyCycleCard import and usage

- **Current**: `MyCycleCard` is imported at line 13 of `calendar.tsx` and rendered at lines 250-257, passing `onLogPeriod`, `onEditSettings`, `onViewAllPeriods` — all three redirect to `/health-sync`.
- **After**: Remove the import statement and the entire `<MyCycleCard ... />` JSX block. The `useRouter` import and `router` variable can also be removed since MyCycleCard was its only consumer in this file.
- **Risk**: None. MyCycleCard's component file stays in the codebase for potential future use. No other screen imports it from `calendar.tsx`.

### CONVERT: PeriodLogPanel from inline Animated.View to Modal bottom sheet

- **Current**: `PeriodLogPanel` renders as an inline `Animated.View` (with `FadeIn` + `SlideInDown` from `react-native-reanimated`) within the Calendar's `ScrollView`. It occupies layout space and pushes content below it down. It is controlled by `selectedDay` state — shown when non-null, hidden otherwise.
- **After**: Convert to a `Modal` bottom sheet following the established pattern used by `PeriodLogSheet.tsx` (lines 268-419) and `SOSSheet.tsx` (lines 47-50):
  - Wrap content in `<Modal transparent visible={sheetVisible} animationType="fade" onRequestClose={onClose}>`
  - Backdrop: `<Pressable style={styles.backdrop} onPress={onClose}>` with `MoonColors.overlay` background
  - Sheet container: `<Animated.View>` with `MoonColors.surface` background, `borderTopLeftRadius: Radii.xl`, `borderTopRightRadius: Radii.xl`
  - Handle bar: 40px wide, 4px tall, `MoonColors.accentPrimary + '40'` background, centered
  - Slide animation: `Animated.spring` (toValue 0, damping 20, stiffness 180) on open; `Animated.timing` (toValue 600, duration 280) on close
  - `KeyboardAvoidingView` wrapping the sheet for note input
  - `maxHeight: '85%'` to prevent sheet from covering the entire screen
- **Content preserved**: All existing PeriodLogPanel sections remain inside the sheet — header (date + cycle day badge), flow intensity selector, symptom chips, tag pills, notes input, save button, delete + cancel row.
- **Animation change**: Replace `react-native-reanimated` (`FadeIn`, `SlideInDown`) with RN core `Animated` spring/timing to match PeriodLogSheet and SOSSheet patterns.
- **New prop**: Add `visible: boolean` prop. Remove reliance on parent conditional rendering (`{selectedDay && <PeriodLogPanel ... />}`).

### MODIFY: CalendarTab handleDayPress — two-tap logic

- **Current** (line 141-143): `handleDayPress` toggles `selectedDay` — sets it to the tapped date or clears it if already selected. This opens/closes the inline PeriodLogPanel for any tapped day.
- **After**: Replace with three-branch logic:
  1. **Future date** (`selected > today`): No action. Return immediately. No visual feedback.
  2. **Unlogged past/today date** (no entry in `periodDayLogs[dateString]`): Call `savePeriodDayLog(dateString, 'medium', [], undefined)` directly. This is the "first tap marks period" behavior. Show success toast. Do not open sheet.
  3. **Already-logged date** (`periodDayLogs[dateString]` exists): Set `selectedDay` and set `sheetVisible = true` to open the bottom sheet for editing.
- **New state**: Add `sheetVisible: boolean` state (default `false`), controlled separately from `selectedDay`.
- **Dependency**: Uses existing `savePeriodDayLog` store action (already imported via `useAppStore`). Uses existing `periodDayLogs` store state (already subscribed at line 23).

### KEEP: Unchanged components

| Component | Reason |
|-----------|--------|
| `CalendarDayCell` | Tap event is handled at parent level; cell rendering logic unchanged |
| Legend (inline `LegendItem`) | All 5 legend items remain |
| `PredictionWindowCard` | Kept in same position (now directly after legend/hint, before scroll bottom) |
| `SaveToast` | Reused for both first-tap success and sheet save success |
| Tap hint text | `tapToLog` / `firstTimeHint` shown when `periodLogs.length === 0` and no day selected |

### BEFORE

```

  ScrollView
    [1] Header (title + subtitle)
    [2] Calendar grid (react-native-calendars with CalendarDayCell)
```

- Net vertical content reduction: ~200px from MyCycleCard removal + ~0px shift from panel-to-modal conversion (modal does not occupy layout space).

## 4. Edge Cases

### Empty state (no logs)

- **Behavior**: Tap hint ("Tap any day to log your period") shows below the legend. All past/today days are tappable and will trigger first-tap-marks behavior (auto-mark with medium flow).
- **Sheet**: Never opens on first tap since the day is unlogged. User would need to tap the same day again (now logged) to open the edit sheet.
- **Risk**: None. Existing `tapToLog` i18n key already describes this interaction.

### Offline / no network

- **First tap**: `savePeriodDayLog` performs an optimistic update (Zustand state updates immediately, Supabase upsert is async best-effort). The day cell will reflect the logged state even offline.
- **Sheet save**: Same optimistic behavior — identical to current PeriodLogPanel save.
- **Risk**: None. Existing pattern handles this.

### Sun role

- **Path**: Sun users hit the `role === 'sun'` guard at line 52 of `calendar.tsx` and return `PartnerCalendarView` (read-only). They never reach the Moon calendar code.
- **Impact**: Zero. No changes to PartnerCalendarView or its dependencies.

### Long press

- **Not implemented**: Only standard `onPress` is used on `CalendarDayCell`. No `onLongPress` handler exists or is planned.
- **Risk**: None.

### Rapid double-tap on unlogged day

- **Scenario**: User taps an unlogged day twice quickly. First tap triggers `savePeriodDayLog`. Second tap fires before state updates and may also call `savePeriodDayLog`.
- **Mitigation**: `savePeriodDayLog` uses Supabase upsert (idempotent). Second call overwrites with identical data. No duplicate records. Alternatively, the implementation can add a brief loading guard (`isSaving` state) to debounce.

### Tapping today

- **Behavior**: Today is a valid target. If unlogged, first-tap marks it. If logged, opens sheet. Same as any past day.
- **Risk**: None.

### Month navigation

- **Behavior**: `handleMonthChange` calls `loadPeriodDayLogs(startDate, endDate)` for the new month. This is unchanged.
- **Sheet state**: If sheet is open during month swipe, it should auto-close (set `sheetVisible = false` in `handleMonthChange`).

### Day within predicted period (not logged)

- **Behavior**: Predicted period days have markers but no `periodDayLogs` entry. First tap marks them as logged. This is correct — predictions are not logs.

## 6. Design Conflicts

**None identified.**

All patterns required for this change already exist in the codebase:
- Modal bottom sheet: `PeriodLogSheet.tsx`, `SOSSheet.tsx`, `WhisperSheet.tsx`
- Optimistic day logging: `savePeriodDayLog` store action

## 7. Accessibility Notes

- Bottom sheet `Modal` has `onRequestClose` for Android back button dismissal (existing pattern).
- Backdrop `Pressable` allows tap-to-dismiss (existing pattern).
- Save/Delete/Cancel buttons retain `activeOpacity` and `accessibilityRole` from current PeriodLogPanel.
- First-tap-marks interaction provides haptic feedback (`impactMedium`) and visual toast — no silent state changes.
