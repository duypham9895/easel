## 1. Current Behavior

### 1.1 Calendar Screen (`app/app/(tabs)/calendar.tsx`)

The calendar view uses the `react-native-calendars` `Calendar` component with its default day cell rendering. Cycle data is displayed through the library's built-in `markedDates` API using `markingType="dot"`:

- **Logged period days** render as solid pink dots/backgrounds (`Colors.menstrual` at `loggedPeriodOpacity`).
- **Predicted period days** render as translucent pink dots/backgrounds (`Colors.menstrual` at `predictedPeriodOpacity`).
- **Fertile window** renders as blue-tinted dots (`Colors.follicular`).
- **Ovulation day** renders as a golden dot (`Colors.ovulatory`).
- **Today** is highlighted with `Colors.textPrimary` as the selected background.
- A color legend sits below the calendar explaining each marker type.
- Tapping a day opens a `DayDetailSheet` (read-only modal) showing the phase name, phase tagline, mood description, and a self-care tip. An "Edit" button on this sheet opens the `PeriodLogSheet`.

Day cells are **not custom-rendered** — they use the library's default circular selection style with no flow-intensity indicators, no phase-gradient backgrounds, and no Flo-style visual treatment.

### 1.2 PeriodLogSheet (`app/components/moon/PeriodLogSheet.tsx`)

A bottom-sheet modal (627 lines) triggered from the `DayDetailSheet` edit button. It provides:

- **Date header** with formatted display of the selected date.
- **Binary period toggle**: either a "Log Period Start" button (when no active period on that date) or an "End Period Here" button (when within an active period with no end date). This is a **single start/end entry per period** — there is no concept of toggling individual days on or off.
- **Override tags**: 5 factor chips — stress, illness, travel, medication, other — used to weight the prediction algorithm. These are **not symptom chips** (no cramps, fatigue, headache, etc.).
- **Notes field**: free-text, 200-character limit.
- **Delete button**: removes the entire period record.
- **Confirmation via `Alert.alert()`**: all actions (start, end, delete) use the native iOS/Android alert dialog. There is no inline toast notification.

### 1.3 PeriodStartButton (`app/components/moon/PeriodStartButton.tsx`)

A dashboard-level shortcut button visible when Moon is near her predicted period window (late luteal, during menstrual, or shortly after). It toggles between "Period Started" and "Period Ended" states. Confirmation uses `Alert.alert()`.

### 1.4 Data Model

The `period_logs` table (migration 001 + 007):

```sql
period_logs (
  id          UUID PRIMARY KEY,
  user_id     UUID NOT NULL REFERENCES profiles(id),
  start_date  DATE NOT NULL,
  end_date    DATE,
  notes       TEXT,
  tags        TEXT[] DEFAULT '{}',  -- override tags only: stress/illness/travel/medication/other
  created_at  TIMESTAMPTZ,
  UNIQUE (user_id, start_date)
)
```

Key constraints:
- `tags` is limited to `['stress', 'illness', 'travel', 'medication', 'other']` via a CHECK constraint.
- One row per period (start_date + optional end_date range), **not** one row per day.
- No `flow_intensity` column exists.
- No `symptoms` column exists on `period_logs`. Symptoms live only in `daily_logs.symptoms` (a separate table for mood/energy/symptom tracking, unrelated to period logging).

TypeScript types:
- `PeriodRecord`: `{ startDate, endDate?, tags? }` — no flow or symptom fields.
- `OverrideTag`: `'stress' | 'illness' | 'travel' | 'medication' | 'other'`

### 1.5 Prediction Engine (`app/utils/cycleCalculator.ts`)

Uses a weighted average of the last 7 cycle gaps. Tagged cycles receive 0.5x weight; recent cycles receive 2x weight. Predictions are rendered as translucent calendar markers. The engine operates on `PeriodRecord` (start/end ranges) and has no awareness of per-day flow intensity or symptoms.

## 2. Desired Behavior

### 2.1 Calendar View — Flo-Style Day Cells

Replace the default `react-native-calendars` day cell rendering with custom `dayComponent` cells:

- **Deep pink (`#D4537E`)** background fill for logged period days.
- **Light pink** (translucent) background fill for predicted period days.
- **Blue** background fill for fertile window days.
- **Bright blue dot** indicator for ovulation day.
- **Flow intensity dot** inside each logged period day cell, sized proportionally:
  - Spotting: small dot (~4px)
  - Light: medium dot (~6px)
  - Medium: larger dot (~8px)
  - Heavy: large dot (~10px)
- Connected-range visual treatment (first day rounded left, last day rounded right, middle days rectangular) for multi-day period spans.

### 2.2 Tap-to-Toggle Period Days

Tapping any calendar day toggles it as a period day:

- **Tap an unlogged day** within a reasonable window (today or past 30 days): marks it as a period day, opens an inline detail panel for flow intensity and symptoms.
- **Tap a logged period day**: opens the detail panel for editing or un-toggles it to remove the log.
- **Multi-day selection**: consecutive tapped days form a contiguous period range automatically. The system should handle gaps intelligently (e.g., tapping day 1 and day 3 does not auto-fill day 2 — each day is individually toggled).

### 2.3 Per-Day Detail Panel

When a period day is selected, show an inline or bottom-sheet detail panel with:

- **Flow intensity selector**: four options — Spotting, Light, Medium, Heavy — with visual radio/chip UI. Default: Medium.
- **Symptom chips** (multi-select): Cramps, Fatigue, Headache, Bloating, Mood swings, Nausea. These are separate from the existing override tags (stress/illness/travel/medication/other), which should remain as "Factors affecting prediction."
- **Per-day notes field**: free-text, 200-character limit (same as current).

### 2.4 Toast Confirmation

Replace `Alert.alert()` confirmation dialogs with inline toast notifications on save/delete actions. The toast should:

- Appear at the top or bottom of the screen.
- Auto-dismiss after ~3 seconds.
- Show a success message (e.g., "Period day logged" / "Period day removed").
- Include haptic feedback (existing `notificationSuccess()` pattern).

### 2.5 Predicted Cycle Overlay

Continue rendering predicted cycle days on the calendar using the existing `cycleCalculator.ts` engine, but with the new Flo-style visual treatment (light pink fill instead of translucent dots).

### 2.6 Partner Real-Time Sync

Sun (boyfriend) should see Moon's logged period data in real-time via Supabase Realtime, consistent with the existing `PartnerCalendarView` pattern. The new per-day detail (flow intensity, symptoms) should be visible read-only to Sun if calendar sharing is enabled.

## 3. Delta Statement

This change transforms the period logging experience from a **binary start/end range model** with override tags and native alert confirmations into a **per-day granular logging model** with flow intensity tracking, symptom chips, custom Flo-style calendar cells, tap-to-toggle day selection, and toast-based confirmations. It requires:

1. **Data model expansion**: new columns (`flow_intensity`, `symptoms`) on a new or restructured per-day period table, replacing the current single-row-per-period `period_logs` model.
2. **Custom calendar rendering**: replacing default `react-native-calendars` day cells with a custom `dayComponent` supporting phase-colored fills, flow-intensity dots, and connected-range visuals.
3. **New UX flow**: tap-to-toggle replaces the current two-step (DayDetailSheet -> PeriodLogSheet) interaction, with inline detail panels for flow and symptoms.
4. **Confirmation pattern change**: native `Alert.alert()` replaced with toast notifications across all period logging actions.
5. **Existing functionality preserved**: override tags, prediction engine, partner sync, and the `PeriodStartButton` dashboard shortcut remain functional.

---

## 4. Business Justification

- **Competitive parity**: Flo, Clue, and other leading cycle apps offer per-day flow logging and symptom tracking. The current binary start/end model feels dated and limits the data richness users expect.
- **Data depth for AI personalization**: per-day flow and symptom data feeds the AI personalization pipeline (CR_20260308_001), enabling phase-aware insights like "Your cramps tend to peak on day 2" and more empathetic partner guidance.
- **User retention**: a richer logging experience increases daily engagement. Users who log symptoms daily have higher 30-day retention in competing apps.
- **Partner value**: Sun seeing flow intensity and symptoms (read-only) provides actionable context beyond "she's on her period," improving the core empathy value proposition.

2. Logged period days display with deep pink (`#D4537E`) filled backgrounds; predicted period days display with light pink translucent fills.
3. Fertile window days display with blue fills; ovulation day displays with a bright blue dot indicator.
4. When a period day is selected, a detail panel presents a flow intensity selector with four options: Spotting, Light, Medium, Heavy.
5. Flow intensity is visually represented on calendar day cells as proportionally sized dots (4px/6px/8px/10px for spotting/light/medium/heavy).
6. When a period day is selected, the detail panel presents multi-select symptom chips: Cramps, Fatigue, Headache, Bloating, Mood swings, Nausea.
7. Each period day supports an individual notes field (200-character limit).
8. Saving a period day shows an inline toast notification (auto-dismiss ~3 seconds) instead of a native `Alert.alert()` dialog.
9. Deleting a period day shows an inline toast confirmation instead of a native alert.
10. The existing override tags (stress, illness, travel, medication, other) remain available as "Factors affecting prediction" and continue to feed the prediction engine.
11. The `PeriodStartButton` on MoonDashboard uses toast confirmation instead of `Alert.alert()`.
12. Sun can view Moon's per-day period data (flow intensity, symptoms) in read-only mode via `PartnerCalendarView` when calendar sharing is enabled.
13. New per-day period data syncs to Sun in real-time via Supabase Realtime.

16. All new UI text is translated in both English (`i18n/en/`) and Vietnamese (`i18n/vi/`).

## 9. Open Questions

### Data Model

1. **[DECISION NEEDED]** Should the data model shift to **one row per period-day** (new `period_day_logs` table) or add per-day detail as a **JSONB column** on the existing `period_logs` table (e.g., `days: { "2026-03-01": { flow: "heavy", symptoms: ["cramps"] } }`)? A per-day row model is cleaner for querying and RLS but requires a new table + migration of existing data. A JSONB approach preserves the current table structure but complicates queries and partial updates.

2. **[DECISION NEEDED]** Should the existing `period_logs` table be **replaced** by the new per-day model, or should both coexist (per-day table for detail, existing table for period ranges)? Coexistence adds complexity but provides a safer migration path.

3. **[DECISION NEEDED]** Should the symptom list (Cramps, Fatigue, Headache, Bloating, Mood swings, Nausea) be stored in the same `symptoms` column pattern as `daily_logs`, or use a separate column/type to distinguish period-specific symptoms from daily check-in symptoms?

### UX

4. **[DECISION NEEDED]** When Moon taps a day to toggle it as a period day, should the detail panel (flow + symptoms) appear **inline below the calendar** (Flo-style) or as a **bottom sheet** (current pattern)? Inline is more fluid but requires calendar scroll management; bottom sheet is consistent with existing UX.

5. **[DECISION NEEDED]** Should tapping a non-period day that falls between two logged period days **auto-fill the gap** (treat as a contiguous range) or remain individually toggled? Auto-fill is more convenient but may cause accidental logging.

6. **[DECISION NEEDED]** Should the `DayDetailSheet` (read-only phase info) be **removed** now that tapping a day directly opens the period toggle, or should it remain accessible via a secondary gesture (e.g., long-press)?

### Partner Sync

7. **[DECISION NEEDED]** Should Sun see **flow intensity labels** (e.g., "Heavy") or only the **visual dot sizing** on the calendar? Explicit labels are more informative but Moon may find them too revealing. This ties into the privacy model.

8. **[DECISION NEEDED]** Should Sun see **individual symptom chips** or only a **summary count** (e.g., "3 symptoms logged")? Similar privacy consideration.

### Toast Library

9. **[DECISION NEEDED]** Should we use an existing React Native toast library (e.g., `react-native-toast-message`, `burnt`) or build a lightweight custom toast component? A library is faster to ship; a custom component matches the Moon theme more precisely.
